# Emit excluded parent directories for re-included `.dockerignore` descendants

## Summary

Fixes Apple/container build-context parity for `.dockerignore` negation patterns that re-include descendants below excluded parent directories.

## Motivation

BuildKit validates that each streamed file arrives after its parent directory. The previous shim-side filter dropped excluded directories while still allowing later negation patterns to re-include descendants. That made valid Docker contexts fail with `changes out of order` during context load.

## Implementation

- Wrap the unpacked build-context cache in `fsutil.NewFilterFS` from the `DiffCopy` path.
- Preserve raw `exclude-patterns` metadata as a string slice so patterns containing commas are not split.
- Remove the hand-rolled `patternmatcher` filtering from the raw `Walk` tar receive path.
- Preserve fork-specific staged Dockerfile behavior by appending negation patterns for requested synthetic Dockerfile paths before creating the `fsutil` filter.
- Keep synthetic Dockerfile request metadata behavior unchanged so the host still receives `dir-name=context` when BuildKit requests the staged Dockerfile.

## Compatibility

This change is behavior-compatible with Docker's `.dockerignore` semantics and only affects the builder shim's context stream ordering/filtering. The macOS host remains unaware of `.dockerignore`; it continues to enforce the context boundary and tar only requested paths.

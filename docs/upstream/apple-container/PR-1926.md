## Motivation And Context

`container exec` can leave an ad hoc process behind when the XPC
client disconnects abruptly, such as after an SSH drop, terminal close,
or crashed caller. The stale process can make later `exec`, `stop`, and
Compose lifecycle operations appear hung or inconsistent.

The useful upstream shape from `apple/container#1926` is the session
ownership boundary: the API server already has an `XPCServerSession`
per connection, so the process-creation route should keep that session
and register process cleanup against it.

## What Changed In The Fork

- Route `containerCreateProcess` directly to the harness so it receives
  the `XPCServerSession`.
- Change `ContainersHarness.createProcess` to accept that session.
- Register `session.onDisconnect` only after the runtime successfully
  creates the requested process and the client attached stdout or
  stderr.
- Kill the specific attached ad hoc process with `SIGKILL` when the
  owning client connection disappears.
- Preserve detached exec process lifetime so `container exec -d`
  continues running after the CLI exits.
- Debug-log cleanup failures because the process may already have
  exited normally.

## Upstream Delta Decision

`apple/container#1926` contains two commits:

1. PTY/socket cleanup when the XPC client disconnects.
2. Stop-timeout enforcement with a parallel force-kill path.

The fork incorporated the first commit's cleanup boundary. The second
commit was reviewed but not copied in this slice because it introduces a
parallel cleanup path that can race normal graceful-stop ownership. A
future stop-timeout slice should preserve a single owner for graceful
stop, forced kill, and cleanup.
